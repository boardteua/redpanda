# T58 — QA: творець кімнати як модератор повідомлень у межах кімнати

## Рішення

- У `rooms` уже є `created_by_user_id` (**T54**). **T58** додає політику **`ChatMessagePolicy`**: творець кімнати може **`PATCH` / `DELETE` чужі** публічні повідомлення **лише** у кімнаті, де він творець; **не** може змінювати пости авторів з `canModerate()` або `isChatAdmin()`.
- Власні повідомлення творця лишаються під правилами VIP / вікна редагування, як і раніше.
- **`RoomResource`:** `created_by_me`, `is_room_moderator` (зараз те саме, що творець).
- **Vue (`ChatRoom.vue`):** `inferCanEditForMessage` / видалення узгоджені з бекендом для WS/мерджу без `can_*` з API.

## Матриця (очікування)

| Актор        | Кімната      | Цільове повідомлення | Очікування `PATCH`/`DELETE` |
|-------------|--------------|----------------------|-----------------------------|
| Творець     | своя         | чужий звичайний юзер | дозволено                   |
| Творець     | своя         | чужий мод/адмін      | 403                         |
| Творець     | чужа / legacy| чуже                 | 403 (як раніше)             |
| Глобальний мод | будь-яка  | не-адмін автор       | дозволено (як **T21**)      |

## Автоматичні перевірки

```bash
cd backend && php artisan test --filter=RoomCreatorMessageModerationApiTest
cd backend && php artisan test
cd backend && npm run build
```

**Статус:** PASS — див. вивід команд у коміті / CI.

## Ручний сценарій (опційно)

1. Зареєстрований користувач з правом створення кімнати створює кімнату.
2. Інший користувач пише в цій кімнаті.
3. Творець бачить кнопки редагувати/видалити на чужому повідомленні і успішно застосовує дію.
4. Повідомлення від модератора/адміна в тій самій кімнаті — дії недоступні (403).
