## Підсумок

Основний технічний ризик для підтримки — **монолітний `ChatRoom.vue` (~2.8k рядків)** без Vuex: логіка кімнати, Echo, приват, друзі/ігнори та UI змішані в одному SFC. ## Backend (`backend/app`)

### Сильні сторони

- Чітке розділення: **Policies**, **Form Requests**, **API Resources**, **Events** для broadcast.
- **Slash-команди** — реєстр + контракти (`App\Chat\SlashCommands\`), що полегшує точкові зміни модерації.
- **Private / friend / ignore** — окремі контролери, throttle у маршрутах.
- Після **T98** — більш передбачувані відповіді при збоях upload (логі, `ensureDirectoryExists`).

### Розмір і складність

| Об’єкт | Рядків (орієнтовно) | Коментар |
|--------|---------------------|----------|
| `Http/Controllers/Api/V1/ChatMessageController.php` | ~470 | CRUD + slash + інлайн-приват + автомод — кандидат на витяг сервісів (`ChatMessageService` / `RoomMessageFlow`) у **окремому Txx**, не в ad-hoc рефакторингу |
| `Http/Controllers/Api/V1/StaffUserController.php` | ~540 | Багато дій staff; `staffUserPayload` централізує форму — добре; далі — модулі за доменом (mute/kick/profile) за потреби |

### N+1 та продуктивність

- **`ChatMessageController::index`** повертає колекцію без `with()` — поля повідомлення денормалізовані в таблиці `chat` (**`post_user`**, **`avatar`** тощо), тож класичного N+1 до `users` у стрічці немає.
- **`ChatMessageResource`** викликає **`Gate::allows('update'|'delete')` на кожен рядок** — для великих `limit` це багато перевірок політик. Laravel кешує завантажені здібності користувача, але вартість залишається **O(n)** на повідомлення. **Рекомендація (P2):** батчеве обчислення прав (наприклад, один прохід по ролі/модерації кімнати) або спрощення політики для списку vs деталі.

- **`PrivateMessageController::conversations`**: для кожного peer викликається **`countUnreadIncoming`** — потенційно **N запитів** до БД на список розмов. **P1:** агрегувати unread одним запитом / підзапитом або кешувати лічильник (обережно з консистентністю).

### Узгодженість з OpenAPI

- У файлі **`docs/chat-v2/openapi.yaml`** присутні шляхи на кшталт **`/api/v1/rooms/...`**, **`/api/v1/images`**, **`/api/v1/mod/...`**, auth, landing тощо.
- У **вступному тексті** OpenAPI описані **`/api/v1/private/...`**, **`friends*`**, **`ignores*`**, **`users/lookup`**, але **відповідні `paths:` для `private/peers`, `private/conversations`, повного набору `friends` / `ignores` не вивірені в цьому аудиті як повне покриття** (швидка перевірка: немає рядків `paths` для `private/peers` у тілі YAML). **P1:** додати операції та схеми відповідей, щоб OpenAPI відповідав `routes/api.php`.

### Політики та безпека (огляд)

- Маршрути захищені **`auth:sanctum`**, **`RejectDisabledAccount`**, throttle-групами — відповідає очікуванню для чату.
- Детальний пентест/Pentest-чекліст виходить за межі T99; опційні **T19** (аватари) лишаються в дорожній карті.
