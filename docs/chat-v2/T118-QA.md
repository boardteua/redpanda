# T118 — Threads / TikTok у стрічці (QA)

## Діагностика (що було не так)

1. **Threads** (`threads.com` / `threads.net`): локальний резолвер підставляв iframe на `threads.net/embed/post/…`. У браузері сторінка відповідає **`X-Frame-Options: deny`** (або еквівалент), тому iframe показує порожній / `chrome-error://…` — це очікувана політика провайдера, не баг мережі.
2. **TikTok short links** (`vm.tiktok.com`, `vt.tiktok.com`, …): реєстр oEmbed (iamcal) для TikTok містить схеми лише для **`https://www.tiktok.com/*`**. Запит **`GET /api/v1/oembed?url=https://vm.tiktok.com/…`** не знаходив endpoint → **422** «Неможливо завантажити вбудовування…».
3. **Кілька посилань в одному повідомленні**: парсер емітить **окремі сегменти** на кожен URL. Якщо в тексті є і Threads, і TikTok, у DevTools буде oEmbed лише для TikTok — це нормально; для одного повідомлення **лише Threads** запиту oEmbed бути не повинно.

## Поведінка після фіксу

- **Threads (пост за шаблоном `…/post/{id}`):** сегмент **`threadsCard`** — картка з посиланням і коротким поясненням, **без iframe** і без виклику oEmbed.
- **TikTok (піддомени `*.tiktok.com`, крім уже покритих `www` / apex у JSON):** бекенд мапить URL на офіційний endpoint `https://www.tiktok.com/oembed`; далі — як для інших oEmbed (санітизація iframe, білий список `allowed_iframe_hosts`). Якщо апстрім TikTok відмовить — **502**; клієнт показує звичайне посилання (як і раніше при помилці запиту).
