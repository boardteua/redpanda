# T176 — Руда Панда LLM (Gemini): продуктова специфікація, ролі, політики даних/безпеки

## Мета (MVP)

- Додати **діалоговий шар** поверх існуючого системного бота (T149–T151): бот **слухає кімнату** та **інколи** відповідає в стрічці.
- Гарантувати **керованість**, **не-флуд**, **приватність**, **безпеку** (API key лише на бекенді), і **предбачувану вартість** (routing моделей + ліміти).

## Невизначеність / юридичні зауваги (не блокують MVP-старт)

Цей документ задає технічні рамки. Остаточні юридичні формулювання (notice/consent) та рішення щодо paid/unpaid — окремим погодженням, але MVP імплементація має бути сумісною з вимогами нижче.

Корисні посилання:

- Gemini API overview: `https://ai.google.dev/gemini-api/docs/api-overview`
- Terms: `https://ai.google.dev/gemini-api/terms`
- Billing: `https://ai.google.dev/gemini-api/docs/billing`
- Rate limits: `https://ai.google.dev/gemini-api/docs/rate-limits`
- Safety settings: `https://ai.google.dev/gemini-api/docs/safety-settings`

## Вмикання/вимикання (глобально й по кімнаті)

- **Master switch (глобально, адмін)**: якщо вимкнено — **жодних** LLM-відповідей ніде.
- **Room-level switch (адмін кімнати)**: якщо вимкнено — у цій кімнаті бот **не** генерує діалогові відповіді.
- **T198:** при глобальному OFF UI редагування кімнати **не** показує блок кімнатного LLM; `PATCH …/rooms/{id}` з `ai_bot_enabled: true` повертає **422** (збережене в БД `false` лишається дозволеним).
- **Системні оголошення** (T149–T151) не є LLM-генерацією. Якщо продуктово допускаємо, вони можуть лишатися активними навіть при вимкненому room-level LLM (погодити в T184; дефолт: **дозволено**, бо це інший режим).

## Матриця ролей (VIP / зареєстрований / гість)

### Відображення LLM-повідомлень

- **VIP**: бачить LLM-відповіді.
- **Зареєстрований**: бачить LLM-відповіді.
- **Гість**: бачить LLM-відповіді (як звичайні повідомлення в кімнаті), якщо кімната не приватна/обмежена іншими правилами доступу.

### Ініціювання / можливості

- **VIP**:
  - Текстові відповіді: ✅
  - Генерація зображень: ✅ (T181), з safety та rate limits
  - Доступ до “важчої” моделі: ✅ (T180)
- **Зареєстрований**:
  - Текстові відповіді: ✅
  - Генерація зображень: ❌ (або ✅ лише після окремого рішення; дефолт MVP: ❌)
  - Доступ до “важчої” моделі: ❌ (дефолт MVP: лише flash/cheap tier)
- **Гість**:
  - Текстові відповіді: ✅, але з консервативнішими лімітами (щоб не стимулювати бот-фарм)
  - Генерація зображень: ❌
  - Важча модель: ❌

## Коли бот відповідає (тригер + cooldown)

Бот **не** відповідає на кожне повідомлення. Мінімальні правила MVP:

- **Пряме звернення**: якщо повідомлення містить явний виклик/згадку (узгодити синтаксис: наприклад, “панда”, “руда панда”, “@panda”, або команда “/panda …”).
- **Запитання до кімнати**: якщо повідомлення є запитанням і має “високу корисність” (евристика; деталі — T180).
- **Icebreaker** при простої: окрема функція T183.

Анти-флуд / cooldown (MVP-політика; точні значення — knobs у T184):

- Не більше **N** LLM-повідомлень на кімнату за **T** хв.
- Мінімальний інтервал між відповідями бота в кімнаті (cooldown).
- Якщо в черзі накопичився backlog або Gemini повертає 429 — degrade gracefully (див. нижче).

## Формат відповіді (натуральний чат)

MVP-вимога (щоб не виглядало як “стаття” і не захаращувало стрічку):

- Один пост = **коротка** відповідь, **1 абзац**, без списків/маркерів, без markdown.
- Якщо відповідь потенційно довга — скоротити або поставити **1 уточнююче питання** (T182).
- Мова: **UA за замовчуванням**. Якщо користувач пише EN — відповідати EN; якщо RU — відповідати RU (або UA з м’яким переходом — за продукт-рішенням; дефолт MVP: відповідати мовою запиту).

## “Пам’ять” / контекст

MVP “пам’ять” = **контекстне вікно** + **summary** (T178):

- Джерело правди — повідомлення кімнати (chat posts).
- У контекст для Gemini потрапляє:
  - останні **N** повідомлень кімнати (conversational window),
  - опційно короткий **summary** останньої розмови (rolling summary) коли вікно “вилазить” за token/size.

Не-MVP (явно відкладено):

- Семантична пам’ять / RAG / embeddings (T185).
- Будь-які експерименти з fine-tuning.

## Персона / system prompt (зберігання і версіонування)

Початковий промпт (persona) має бути:

- **System/developer instruction** (не як user message).
- Зберігатися та керуватися через адмінку (T184), з:
  - `revision`/`updated_at` (аудит),
  - лімітом довжини,
  - кнопкою “reset to default”.

Промпт має включати:

- Тон: доброзичливий, короткий, не повчає, не флудить.
- Поведінка при неоднозначності: ставить **1 уточнення** (T182).
- Табу: не розкривати секрети/ключі, не робити вигляд “staff/moderator”, не інструктувати про незаконне.

## Політика даних: мінімізація, PII, логування

### Мінімізація даних у промптах

- Не передавати в Gemini “зайвого”: лише ті повідомлення, які потрібні для відповіді.
- Уникати передачі PII “як є” там, де можливо:
  - не включати email/телефон/адреси в summary,
  - не включати токени/посилання з секретами,
  - у майбутньому: редакт/маскування (не-MVP).

### Логи

- У прод-логах (structured logs) **не** зберігати повний текст користувача або повний `contents`.
- Дозволено логувати метадані: `room_id`, `job_id`, `model_id`, latency, tokens/usage (якщо доступно), тип тригера (direct_mention/icebreaker/etc), статус (ok/timeout/429/blocked).
- Повний текст допускається лише у dev/staging за явною ознакою (конфіг), і лише якщо це не суперечить policy.

### Секрети

- API key Gemini **ніколи** не потрапляє у фронт: тільки `.env` + server-side config.
- Заборонено логувати API key або full request headers.

## Safety / policy enforcement

- Використати **safety settings** Gemini, щоб зменшити ризик небажаного контенту.
- Якщо Gemini блокує відповідь (blocked/safety) — degrade gracefully:
  - або “мовчати” (нічого не постити),
  - або постити 1 короткий нейтральний рядок (“не можу відповісти на це”).
  - Остаточна продуктова політика: вибрати один варіант у T176/T179 (дефолт MVP: **нейтральний рядок**, щоб UX був пояснюваний).

## Надійність і ліміти (429/timeout/backpressure)

- Усі виклики до Gemini — з **черги** (T177), з retry/backoff на 429.
- При системному backpressure:
  - не роздувати чергу нескінченно на одну кімнату,
  - не постити “вибачте” повідомлення на кожну помилку (обмежити частоту).

## Моделі і маршрутизація (чернетка до T180)

MVP-рамка (без прив’язки до конкретних model IDs — вони конфігуруються у T184):

- “Дешева/швидка” модель для звичайних відповідей.
- “Потужна” модель — лише для VIP або для підтверджено складних запитів (T180).
- Image — окрема модель/endpoint (T181).

## Взаємодія з модерацією

- Бот не має “модераторських” прав.
- Якщо в чаті є заборонені теми/слова — правила застосовуються і до бота (наскільки це можливо технічно), або відповіді повинні бути нейтральні/відмовні.
- Узгодження з T12/T53: бот не має підказувати, як обійти бан/фільтри/правила.
