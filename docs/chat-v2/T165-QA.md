## Що перевірено

| Критерій | Результат |
|----------|-----------|
| Збереження browser subscription | `POST /api/v1/push/subscriptions` для зареєстрованого користувача створює/оновлює `push_subscriptions` |
| Гість не може керувати push | `PushSubscriptionApiTest` покриває **403** для guest |
| Ідемпотентне видалення | `DELETE /api/v1/push/subscriptions` повертає `meta.removed` = `1`, повтор — `0` |
| Queue після нового повідомлення | `ChatApiTest` / `PrivateMessageApiTest` перевіряють enqueue `SendWebPushForRoomMessage` / `SendWebPushForPrivateMessage` лише для нового повідомлення, не для дубля |
| Безпечний payload + очистка expired endpoint | Реалізовано в `App\Services\Push\WebPushService`: короткий preview, room/private data, `isSubscriptionExpired()` → delete |
| Мінімальний клієнтський opt-in | `RpWebPushPrompt.vue` викликає `Notification.requestPermission()`, `pushManager.subscribe()`, POST/DELETE на API |
| Мінімальний service worker | `resources/js/sw.js`: `push` → `showNotification`, `notificationclick` → focus/navigate за `data.url` (**T166** — deep link кімната/приват) |

## Ручний smoke для staging / production

1. У `.env` задати `WEB_PUSH_VAPID_SUBJECT`, `WEB_PUSH_VAPID_PUBLIC_KEY`, `WEB_PUSH_VAPID_PRIVATE_KEY`.
2. Працюючі процеси: web + `php artisan queue:work`; для real-time сценарію чату — також `php artisan reverb:start`.
3. Відкрити чат під зареєстрованим користувачем через HTTPS, натиснути кнопку `Увімкнути push`.
4. Із другого акаунта надіслати:
   - приватне повідомлення;
   - публічне повідомлення в доступну кімнату.
5. Очікування:
   - у браузері з’являється системне push-сповіщення;
   - клік по ньому фокусує або відкриває відповідний URL (кімната за slug / приват за `private_peer` — **T166**);
   - у логах немає VAPID секретів; expired endpoint видаляється автоматично.
