# T162 — QA: оптимізації відправки на повільній мережі

**Епік:** T160 (PWA) · **Звіт аналізу:** [T161-SLOW-NETWORK-SEND.md](./T161-SLOW-NETWORK-SEND.md) (оновлено серією B §3.1).

## Автоматичні перевірки

- `cd backend && npm run build` — PASS (виконано при закритті задачі).
- `cd backend && php artisan test` — PASS (регресія API / ідемпотентність `client_message_id` без змін контракту).

## Зміни (узгоджено з беклогом T161)

| Пріоритет | Пункт | Реалізація |
|-----------|--------|------------|
| P0-1 | Менше RTT на CSRF | `ensureSanctum()` пропускає GET, якщо є cookie `XSRF-TOKEN`; відповідь **419** → один повтор після `/sanctum/csrf-cookie` (axios interceptor у `resources/js/bootstrap.js`). |
| P0-2 | Приват: не блокувати на списку розмов | Після успішного POST — `void this.loadConversations()` (фон). |
| P0-3 | Optimistic UI | Кімната: тимчасовий рядок (негативний `post_id`) + заміна за `client_message_id`; приват: негативний `id` + та сама схема. Індикатор «відправка…» у `ChatFeedMessageRow` / `PrivateChatPanel`. Без optimistic для `/`-команд і для повідомлень з `image_id`. |
| P0-4 | Черга / WS | Коментар у `backend/.env.example` біля `QUEUE_CONNECTION` з посиланням на T161 (P0-4). |
| P1-2 | Retry при збої мережі | Один повтор POST з тим самим `client_message_id` при типових мережевих помилках axios (без відповіді / `ERR_NETWORK` / `ECONNABORTED`). |
| P1-3 | Індикатор | Прапорець `rp_send_pending` на optimistic рядках; існуючі `sending` / `sendingPrivate` без змін. |

## До/після (метрика серії B)

Порівняння послідовних HTTP RTT — за тією ж методикою, що в **T161** (Chrome throttling, **Preserve log**). Очікувані зміни зафіксовані в **T161** §3.1; для числових ms (серія A) знімок у DevTools на цільовому середовищі — за шаблоном таблиці в T161 §3.

## Регресія (ручний smoke)

- Кімната: текст, редагування, повідомлення з зображенням (без optimistic прев’ю в стрічці — як раніше).
- Приват: відправка, список розмов оновлюється після фонового `loadConversations`.
- Повтор після обриву мережі: той самий `client_message_id` не створює дубль на сервері (ідемпотентність T04).
