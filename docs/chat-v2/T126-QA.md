# T126 — сайдбар «Люди»: без миготіння «онлайн» після refresh (QA)

## Канонічний шлях (варіант A з чекліста)

- До підтвердженого мапінгу `user_id → presence` з REST / подальшого синку показується семантика **«невідомо»** (`PRESENCE_STATUS_UNKNOWN`), візуально узгоджена з **офлайн / неактивний** (без зеленого «хибного онлайн»).
- Після гідрату (`peerPresenceStatusByUserId`, Echo `here()` тощо) застосовуються реальні статуси (**T20** / **T48**).

## Реалізація (код)

- **`chatSidebarPresence.js`:** константа `unknown`, `presenceRowClass` / `presenceDotClass` для `unknown` → ті самі класи, що для `inactive`; підпис «уточнюється» у `presenceLabelUa`.
- **`ChatRoom.vue`:** прапорець **`peerPresenceStatusFetchLoading`** — під час початкового завантаження статусів пірів панель отримує `unknown` для рядків без мапінгу.
- **`ChatSidebarUsersPanel.vue`:** `peerSidebarStatus()` повертає `PRESENCE_STATUS_UNKNOWN`, коли `peerPresenceStatusFetchLoading === true` і для `p.id` ще немає запису в `peerPresenceStatusByUserId`.
- **`chatSidebarPresence.motion.css`:** плавні переходи рядка / крапки; `prefers-reduced-motion` враховано.

## Ручна перевірка (hard refresh)

1. Відкрити кімнату чату → вкладка **«Люди»**.
2. **Hard refresh** (Ctrl+Shift+R / Cmd+Shift+R).
3. **Очікування:** інші учасники з HTTP-списку **не** мають короткого зеленого спалаху «онлайн», якщо для них ще немає підтвердженого presence; після швидкого приходу WS/REST статуси оновлюються без оманливого початкового «усі зелені».
4. Два клієнти: один реально офлайн — після refresh **не** показувати його як онлайн до синку.
