# T185 — QA: семантична пам’ять (bounded retrieval без embeddings)

## Вердикт

**PASS** (2026-04-01)

## Докази

### Backend (PHPUnit)

```bash
cd backend && php artisan test tests/Feature/Ai/RoomRetrievalServiceTest.php
```

Очікування: усі тести **зелені**; покрито:

- retrieval працює **лише в межах кімнати** та повертає **ранжовані** фрагменти;
- **не** використовує повідомлення системних ботів як corpus;
- поважає `excludePostId` (не підтягує тригер-пост у підказки);
- поважає retention-вікно `chat.ai_retrieval_window_days`.

## Поведінка (коротко)

- **Що це**: lightweight RAG без векторів/embeddings і без fulltext — простий lexical overlap по токенах, з жорсткими лімітами.
- **Де застосовується**: перед викликом Gemini у `GenerateRudaPandaRoomReplyJob` додається блок:
  - `Коротке зведення контексту кімнати` (T178 summary), і
  - `Релевантні фрагменти…` (T185 retrieval), і
  - останні повідомлення з пам’яті кімнати.
- **Конфіги** (див. `backend/config/chat.php`):
  - `chat.ai_retrieval_window_days` (env `CHAT_AI_RETRIEVAL_WINDOW_DAYS`, дефолт 30)
  - `chat.ai_retrieval_scan_limit` (env `CHAT_AI_RETRIEVAL_SCAN_LIMIT`, дефолт 200)

## Обмеження та ризики

- **Якість**: це не семантичний пошук; можливі хибні спрацювання/нерелевантні підказки. У промпті помічено як “можуть бути неточні”.
- **Privacy / retention**: retrieval обмежений часовим вікном і не тягне весь архів; системні боти виключені з corpus.
- **Продуктивність**: лише останні `scan_limit` повідомлень у межах вікна, без важких індексів/векторних сторажів.
