# T83 — Відкат після деплою

Коротка процедура для операторів. Деталі деплою — [T80-DEPLOY-CHECKLIST.md](./T80-DEPLOY-CHECKLIST.md).

## Коли відкатувати

- Помилки після релізу (5xx, зламані міграції, некоректна збірка фронту).
- Некоректні зміни схеми БД, які не можна виправити швидким hotfix.

## Варіант A — код і образи (без відкату БД)

1. Повернути репозиторій на попередній **tag** або **commit** (`git checkout <ref>`).
2. Повторити кроки деплою з T80: `composer install`, `npm ci && npm run build`, **`php artisan migrate`** лише якщо міграції сумісні назад (зазвичай **не** запускати `migrate:rollback` у проді без узгодження).
3. Перезапустити **PHP-FPM / контейнери**, **queue worker**, **Reverb**.
4. Перевірити `GET /health/ready` і короткий сценарій real-time (два клієнти в кімнаті).

Підходить, якщо БД не змінилась критично або міграція idempotent / зворотна.

## Варіант B — відновлення БД з бекапу

Якщо міграція або додаток зіпсували дані:

1. Зупинити трафік (балансувальник / `php artisan down` за політикою команди).
2. Відновити останній **успішний** dump (наприклад `gunzip -c backups/redpanda-….sql.gz | mysql …`).
3. Вирівняти **версію коду** з тим станом, який очікує ця БД (той самий tag/commit, що був до проблемного деплою).
4. Підняти сервіси, `php artisan up`, перевірити health і smoke-тести.

**Увага:** втрата даних після моменту бекапу — прийняти явно з зацікавленими сторонами.

## Варіант C — лише фронт

Якщо зламана лише Vite-збірка:

1. Перезібрати з коректними `VITE_*` або відкотити артефакт `public/build` з попереднього релізу.
2. Очистити CDN/кеш за наявності.

## Пов’язані артефакти

- Шаблон кроків на сервері: `docker/deploy.example.sh` (бекап перед міграціями).
- CI: `.github/workflows/ci.yml` (тести + збірка; SSH-деплой — лише якщо налаштовані `vars.DEPLOY_*` і secrets).
