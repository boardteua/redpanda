# T35 — Безпека поля повідомлення (композер + відображення)

**Статус:** PASS (після перевірки та додаткових тестів довжини/HTML-літералів)

**Обсяг:** текст повідомлення з композера → `POST /api/v1/rooms/{id}/messages`, збереження, JSON API, Vue-стрічка (`ChatFeedMessageRow`), архів (`ArchiveChat`), форматування T30 (`style`), paste зображень T34.

## 1. XSS і HTML у тілі повідомлення

| Перевірка | Висновок |
|-----------|----------|
| Збереження в БД | Текст зберігається як рядок без серверного HTML-рендеру; санітизації HTML на запис **немає** (не потрібна для моделі «plain text»). |
| API | `post_message` у JSON — звичайний рядок UTF-8. |
| Vue | `ChatFeedMessageRow` і архів рендерять тіло через **`{{ message.post_message }}`** (інтерполяція тексту) — Vue екранує `<`, `>`, `"`, що **нейтралізує** типовий XSS у нашому SPA. |
| `v-html` | Для тексту чату **не** використовується. `v-html` у сайдбарі — лише для **статичних** SVG іконок вкладок (`tab.icon`), не з повідомлень користувачів. |

**Ризик для сторонніх клієнтів API:** якщо інший клієнт підставляє `post_message` в `innerHTML` без екранування — це вже відповідальність того клієнта. Поточний продукт (Vue 2 SPA) — безпечний за наведеною схемою.

## 2. Ін’єкція розмітки / «змішаний» paste

- Текст із буфера в textarea лишається текстом; **немає** `contenteditable` з HTML.
- Paste зображення (T34) обробляє лише **file** з дозволеними MIME; інші типи не перехоплюються — текстовий paste не ламається.

## 3. Довжина повідомлення

- Сервер: `StoreChatMessageRequest` — `max` залежить від ролі (гість **2000**, звичайний **4000**, VIP/мод/адмін **8000**). Обхід через прямий API без зміни прав — очікувано; **композер** обмежує `maxlength="4000"` для всіх, тому VIP/мод можуть відправити довший текст лише поза UI (низький ризик, радше UX-розбіжність).

## 4. Обхід санітизації / форматування (T30)

- Поле `style` валідується whitelist (`ChatMessageBodyStyle::BG_KEYS` / `FG_KEYS`, булеві прапорці).
- На клієнті `normalizePostStyleFromApi` повторно відкидає невідомі ключі перед побудовою класів `rp-chat-msg-*` — немає довільних імен класів з даних мережі.

## 5. Модерація (T12)

- `ContentWordFilter` працює в plain text (заміна підрядків), не інтерпретує HTML.

## 6. Вкладення зображення

- Ліміти MIME/розміру на upload (T10/T34); `post_message` + `image_id` — окремі поля; зображення показується як `<img :src="url">` з URL, що видає бекенд (не з довільного тексту повідомлення).

## Рекомендації (не блокери)

- **UX:** узгодити `maxlength` textarea з фактичним `max` для VIP/staff (або прибрати жорсткий maxlength і покладатися лише на відповідь 422).
- **Майбутні клієнти:** документувати в OpenAPI/гайді, що `post_message` — **plain text**, не HTML.

## Докази (автотести)

- `php artisan test` — PASS (повний прогін після змін).
- `ChatApiTest`: літерали з `<`, `&`, лапками зберігаються без змін; перевищення **4001** символу для звичайного користувача та **2001** для гостя дає **422** на `message`.
