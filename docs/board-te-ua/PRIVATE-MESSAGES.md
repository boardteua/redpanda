# Приватні повідомлення (ПП) на board.te.ua

**Оновлено:** 2026-03-20  
**Метод:** Chrome DevTools MCP — кліки по DOM, перегляд мережевих запитів, реальне повідомлення користувачу **org100h**.

---

## Структура повідомлення в загальному чаті

Кожен рядок стрічки — це `li.ch_logs` з атрибутами `data-user`, `data-pid`:

- **Нік:** `span.username` (класи на кшталт `vip`, `sadmin` тощо).
- **Аватар:** `div.my_avatar.chat_avatar_wrap` з `value="нік"` та `img.avatar_chat`.

---

## 1. Клік по нікнейму (згадка / відповідь у загальному чаті)

**Спостереження:** клік по `span.username` у рядку повідомлення вставляє в поле `#content` префікс на кшталт **`нік > `** (приклад для org100h: `org100h > `).

- Поле **`#this_target`** при цьому лишається **`none`** — це все ще **загальний** чат, оформлений як відповідь/згадка в тексті.
- Після дописування тексту повідомлення відправка йде через **`POST /system/chat_process.php`** (як і звичайний пост).

**Тест:** надіслано в загальний чат рядок зі згадкою org100h і текстом про тест звукового сповіщення — повідомлення з’явилось у стрічці.

### Звукові сповіщення (ресурси сторінки)

На завантаженні чату браузер підвантажує кілька файлів у `/sounds/`, зокрема:

- `username.mp3`
- `newpost.mp3`
- `pmsound.mp3`
- `whistle.mp3`

Логіка вибору звуку зашита в клієнтському JS (у цьому огляді код не розбирався). З назв файлів **логічно** припустити окремий сигнал для **згадки за ніком** (`username`) і для **привату** (`pmsound`) порівняно зі звичайним новим постом (`newpost`). Щоб підтвердити поведінку на 100%, варто прослухати в браузері або знайти виклики `Audio` / `play()` у `out.js` / `fut.js`.

## 2. Клік по аватарці (команда `/msg` у тому ж полі)

**Спостереження:** клік по `div.my_avatar` для рядка користувача вставляє в **`#content`** рядок на кшталт **`/msg org100h `** (пробіл у кінці — зручно дописати текст).

- Це **той самий** основний інпут, що й для загального чату (`#main_input`), тобто «один потік вводу».
- У тесті **пряме** надсилання форми через `requestSubmit()`, коли в `content` лишився префікс `/msg …`, призвело до **`POST /system/chat_process.php` з HTTP 500** — тобто текст пішов у **загальний** обробник як звичайний `content`, а не був перехоплений як приват.

**Висновок для користувача:** префікс `/msg` з аватарки розрахований на роботу разом із **клієнтською** логікою (ймовірно, відправка в `private_process.php` або перехоплення перед сабмітом). Якщо відправити форму «сирим» способом без цієї логіки, сервер може помилитись. **Надійний шлях привату** — вікно привату (нижче) або пункт сайдбару **«Приватний чат»**.

## 3. Сайдбар: «Приватний чат» (`li.send_private`)

**Шлях:** `#chat_panel` → вкладка **Люди** (`#chat_user`) → клік по рядку користувача → меню → пункт **«Приватний чат»** (`li.user_option_separator.send_private`, у `value` передається нік адресата).

**Результат:**

1. Відкривається **`#private_panel`** (`display: block`) — плаваюча панель (jQuery UI draggable / resizable).
2. У шапці: нік адресата (клас `private_target`), кнопка додавання в друзі `add_friend_button`.
3. Область історії: `#show_private` всередині `#private_chat` (спочатку може бути «Повідомлення відсутні»).
4. Форма **`#private_input`** (`name="private_data"`):
   - приховані `pv_target`, `pv_hunter` (у тесті `pv_hunter` = поточний користувач);
   - поле **`#private_content`** (`type="text"`, `maxlength="999"`);
   - **`#submit_private`** — відправка.

**Мережа після відкриття:** періодично викликається **`GET /system/private_log2.php?target=органік`**.

**Надсилання:** **`POST /system/private_process.php`** з тілом на кшталт:

- `target=org100h`
- `content=…` (текст повідомлення)
- плюс поля форматування: `bold`, `italic`, `underline`, `high`, `color`

**Тест для org100h:** через меню **send_private** відкрито приват, у поле введено текст вітання та позначку `[doc PM] …`. Після сабміту відповідь **200**, повідомлення з’явилось у `#show_private` з міткою дати/часу.

**Примітка:** одразу після відкриття панелі в DOM поле **`pv_target`** могло бути **порожнім**; природний сабміт через кнопку все одно відправив коректний `target` у тілі POST. У разі ручного виклику сабміту з скрипта інколи знадобилось виставити `pv_target` вручну — це нюанс автоматизації, не обов’язково типова поведінка UI.

**Скріншот:** [screenshots/private-panel-org100h.png](screenshots/private-panel-org100h.png).

---

## 4. Вкладка «Приват» у сайдбарі (`#chat_private`)

Іконка **`#chat_private`** перемикає вміст панелі на список/стан приватів. У порожньому стані показується текст **«Немає нових повідомлень»**. Детальне листування тредів у цьому проходженні не розгорталось; повноцінне вікно переписки — у **`#private_panel`**.

---

## 5. Команда в чаті `/msg` (довідка)

У довідці з сайдбару зазначено: **`/msg ім'я користувача Текст`** — відправити особисте повідомлення. Це узгоджується з префіксом, який вставляє клік по аватарці; фактична маршрутизація на сервер залежить від обробника в JS при натисканні «Відправити» в інтерактивному режимі.

## Підсумок сценаріїв

| Дія | Куди потрапляє текст | Примітка |
|-----|----------------------|----------|
| Клік по **ніку** | `нік > …` у загальний чат | `chat_process.php` |
| Клік по **аватарці** | `/msg нік …` у поле загального вводу | Потрібна клієнтська відправка привату; «голий» сабміт у тесті дав **500** на `chat_process.php` |
| **Приватний чат** у меню користувача | Окреме вікно + `private_process.php` | У тесті **успішно** для org100h |
