# Chat v2 — task list (Agents Orchestrator)

**Правила:** одна задача в роботі; після реалізації — **код-рев’ю** (див. `docs/chat-v2/AGENT-ORCHESTRATION.md`), потім QA з **PASS** лише за доказом (логи, тести, скріншоти для UI). Якщо QA передбачає **live-чат (Echo / WebSocket)** — перед PASS перевірити середовище: **`php artisan reverb:start`**, **`php artisan queue:work`** (при `QUEUE_CONNECTION=database`), узгоджені **`VITE_REVERB_*`**; повний регламент — розділ **«Локальне середовище real-time при завершенні задачі»** у `docs/chat-v2/AGENT-ORCHESTRATION.md`. До **3** спроб на задачу; далі — escalate. Відкриті задачі: рядки `### [ ]`; закриті: `### [x]`.

**Специфікація:** [project-specs/chat-v2-setup.md](../project-specs/chat-v2-setup.md)  
**Оркестрація:** [docs/chat-v2/AGENT-ORCHESTRATION.md](../docs/chat-v2/AGENT-ORCHESTRATION.md)

### [x] T55 — (Опційно / довгостроково) Backend oEmbed: реєстр провайдерів, проксі-ендпоінт, SSRF-захист і кеш (розширення після T46)

- **Delegate:** Backend Architect (Laravel, HTTP-клієнт, безпека, кеш)
- **Залежність:** **T46** (зараз ембеди й лінки будуються на клієнті з жорсткими патернами); ця задача **не** скасовує T46 — дає серверний шлях для провайдерів з реєстру **oEmbed** ([специфікація та реєстр](https://oembed.com/), репозиторій [iamcal/oembed](https://github.com/iamcal/oembed); npm [`oembed-providers`](https://www.npmjs.com/package/oembed-providers) як зручне джерело `providers.json`)
- **Контекст для бекендера:** споживач oEmbed знаходить провайдера за **схемами URL** з реєстру, будує GET на **endpoint конкретного провайдера** з параметром `url` (ресурс) і опційно `maxwidth` / `maxheight` / `format`, отримує JSON (або XML) з полями на кшталт `type`, `html`, `width`, `height`, `thumbnail_url`, `cache_age`. Завдання — безпечно проксувати це в наш API, кешувати відповіді, не піддаватись **SSRF** і не віддавати клієнту сирий небезпечний HTML.
- **Deliverables:**
  - **Реєстр провайдерів:** зафіксувати в репо **версіонований** артефакт (наприклад оновлюваний `providers.json` з npm `oembed-providers` або еквівалент з iamcal) + короткий нотаток у `docs/chat-v2/` *як оновлювати* (без секретів); **алгоритм пошуку провайдера** як у прикладі з реєстру (wildcard `*` у схемах → regex)
  - **HTTP-клієнт Laravel:** вихідні запити лише на **host з обраного endpoint** реєстру (після матчу URL користувача); **жодних** прямих GET на довільний `url` з клієнта як ціль вихідного запиту; таймаут, ліміт розміру тіла відповіді, обмеження редіректів (0 або 1 — узгодити в PR)
  - **Вхідний контракт API:** напр. `GET /api/v1/oembed?url=` (або `POST` з body — узгодити з OpenAPI); валідація: лише `http`/`https`, довжина, **optionally** перевірка що URL матчиться одній схемі з реєстру перед вихідним запитом; **auth:** узгодити з політикою продукту (типово Sanctum SPA — лише автентифіковані, щоб не давати відкритий проксі)
  - **SSRF / зловживання:** відхиляти приватні/лінк-локальні адреси у **вхідному** `url` (RFC1918, loopback, metadata cloud — мінімум той самий рівень обережності, що для webhook-ів); **rate limit** / throttle на ключ користувача або IP; логувати без повного PII у verbose
  - **Кеш:** Redis або `Cache` store — ключ від нормалізованого `url` (і за потреби `maxwidth`), TTL з `cache_age` з відповіді або дефолт з конфігу; заголовки `Cache-Control` для клієнта за бажанням
  - **Відповідь клієнту:** безпечний піднабір полів (`type`, `title`, `author_name`, `thumbnail_url`, `html` лише після санітизації — див. нижче); або **лише метадані + `embed_url`** якщо HTML відкидаємо повністю (узгодити в PR)
  - **Санітизація `html` з oEmbed:** якщо повертаємо `html` — **whitelist** дозволених тегів/атрибутів (типово `iframe` з `src` лише з **дозволених доменів** провайдера: `youtube.com`, `twitter.com`, … — список у коді або винесений у конфіг); прибрати `<script>`, події `on*`, `javascript:`; узгодити з духом **T35** (немає виконання довільного скрипта від імені нашого домену)
  - **Помилки:** 422 якщо URL не матчиться жодному провайдеру; 502/504 якщо провайдер недоступний; не пробивати внутрішні деталі мережі в JSON помилки
  - **Документація:** оновити **`docs/chat-v2/openapi.yaml`**; короткий **`docs/chat-v2/T55-oembed.md`** (потік запиту, SSRF checklist, приклад curl)
  - **Інтеграція з SPA (scope):** мінімум у цій задачі — **стабільний JSON API**; опційно в тому ж PR — виклик з `ChatMessageBody` / парсера **T46** тільки якщо обсяг дозволяє; інакше окремий пункт у чеклісті після T55
- **QA evidence:** PHPUnit (або Pest): успішний шлях з **HTTP fake** на відомий oEmbed endpoint; кейс «URL не з реєстру» → 422; кейс «вхідний url = loopback / приватна мережа» → 422/403; переконатися що **реальний** вихідний запит не йде на довільний host (assert на URL у fake); за наявності санітизатора — тест що `<script>` у фейковій відповіді не потрапляє в JSON для клієнта; `php artisan test` PASS; приклад `curl` у T55-oembed.md

### [x] T76 — **Auth0**: OIDC / **Universal Login** + соціальні провайдери (**Google**, **Facebook**, **X**)

- **Delegate:** Full stack (+ DevOps за потреби DNS / custom domain у Auth0)
- **Залежність:** **T02**/**T03** (логін/пароль і гість **лишаються** в Laravel — додати соц-кнопки та callback-флоу); **T75** (опційні прапорці вітальні); **T21** (ролі після провізіонінгу); **T13** або окремий імпорт — узгодити з **provision** (імпортований юзер + перший вхід через Google тощо)
- **Контекст (Auth0):** [Auth0](https://auth0.com/docs) пропонує **централізований вхід** (**[Universal Login](https://auth0.com/docs/authenticate/login/auth0-universal-login)**) з підтримкою **соціальних провайдерів** (Bring Your Own Identity) і стандарту **[OpenID Connect (OIDC)](https://auth0.com/docs/authenticate/protocols/openid-connect-protocol)** для клієнтських застосунків; знімає частину ризиків «зробити свою» ідентичність і спрощує додавання **кількох** IdP. У задачі — **не** дублювати окремо OAuth у Laravel для кожної мережі, а **підключити провайдерів у Auth0** (Google, Facebook, X) і приймати на бекенді **випущені Auth0 токени** (перевірка підпису через **JWKS**).
- **Рішення клієнта (зафіксовано):** **логін і пароль** залишаються в **Laravel** (**Sanctum** / поточний потік **T02**), бо **велика частина користувачів** буде **імпортована** з legacy — хеші/облікові записи **не** переносити в **Auth0 Database**. **Гість** — **обов’язково** лишається як зараз: **лише** Laravel (**реєстрація гостя / cookie**), **без** Auth0, **без** Hosted Login і **без** JWT; соц-кнопки **не** замінюють і не ламають «Зайти анонімно». **Auth0** — **додатково** для **зареєстрованих** через соцмережі (Google, Facebook, X); після успішного OIDC — **provision** або **прив’язка** до рядка в `users` (унікальність по `auth0_subject` та/або правила **email/нік** — узгодити в PR).
- **Deliverables:**
  - **Auth0 (Dashboard / IaC за політикою команди):** Tenant; **Application** типу **SPA** (Vue **T03**) з коректними **Allowed Callback / Logout / Web Origins**; **API** (Resource Server) з **Identifier** для **audience** access token; увімкнути та налаштувати **Social Connections**: **Google**, **Facebook**, **X** (Twitter); **Universal Login** (брендинг/мова за потреби); опційно ознайомитись з можливостями **[MFA](https://auth0.com/docs/secure/multi-factor-authentication)** та **[Attack Protection](https://auth0.com/docs/secure/attack-protection)** — винести в follow-up, якщо не входить у MVP
  - **Vue 2.7 (T03):** інтеграція **[Auth0 SPA SDK](https://auth0.com/docs/libraries/auth0-spa-js)** (або еквівалент з **PKCE**): кнопки «Увійти через …» → **redirect** на Hosted Login або `loginWithRedirect` з `connection` за потреби; маршрут **callback**; збереження сесії; **logout** з очищенням Auth0; **не** класти **client_secret** у фронт (для SPA використовується лише **public client_id** + PKCE)
  - **Laravel:** **два шляхи авторизації API** без взаємного виключення: (1) існуюча **сесія Sanctum + cookie** для **логін/пароль** і **гостя** (**T02**) — **гість ніколи** не отримує Auth0 `sub` і не проходить через JWT-шар; (2) **Bearer JWT від Auth0** **лише** для сесій після соц-логіну — middleware з **JWKS** (`iss`, `aud`, `exp`). Для (2): **provision** нового `users` або **merge** з імпортованим користувачем за **email** (або іншим ключем) з явною політикою колізій; поле **`auth0_subject`** (унікальне, nullable). **Не** використовувати Auth0 як primary password store
  - **T75:** якщо потрібно керувати видимістю соц-кнопок з адмінки — читати прапорці на **T03**; інакше UI завжди показує всі підключені в Auth0 connections
  - **Документація:** **`docs/chat-v2/T76-QA.md`** + короткий **`docs/chat-v2/T76-auth0-setup.md`**: змінні `.env`, посилання на Auth0 Dashboard кроки, діаграма redirect (без секретів у репо)
- **QA evidence:** ручний прохід **Google / Facebook / X** у staging; **регресія: гість** входить і користується чатом як **до** T76; PHPUnit (mock JWT / fixture keys); `npm run build` PASS; перевірка **403** без валідного токена
- **Негативний scope:** повна **міграція** користувачів з legacy **org100h** у Auth0 — окремий таск; **Auth0 Organizations** / enterprise SAML — не входить без окремого рішення
- **Відкрите питання (PR):** правила **account linking** (соц-логін на вже імпортований email); **Refresh Token** rotation і зберігання (httpOnly cookie vs in-memory) для SPA
